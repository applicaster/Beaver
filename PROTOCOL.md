# LoggerNext — Wire Protocol Contract

This document captures the WebSocket protocol that LoggerNext must speak
with the mobile SDK. The mobile SDK is the source of truth (D2 / D10 note:
we do not change the wire format). Anything inferred from the existing
Logger code is marked as **(inferred — verify with SDK team)**.

If your reading of the SDK disagrees with anything here, the SDK wins and
this doc gets corrected.

---

## 1. Transport

| Property        | Value                                           |
|-----------------|-------------------------------------------------|
| Protocol        | WebSocket (RFC 6455) over plain TCP             |
| Port            | `9080`                                          |
| Address family  | IPv4 and IPv6 (LoggerNext will accept both)     |
| TLS             | Not used                                        |
| Auto-ping       | Server replies to client pings automatically    |
| Subprotocols    | None negotiated                                 |
| Origin          | Not checked                                     |

The current Logger code restricts the listener to IPv4 only. **(inferred —
verify whether the SDK ever connects via IPv6, e.g., on a tethered iPhone
where v6 is preferred.)** LoggerNext will accept both unless the SDK side
forbids it.

---

## 2. Frame format

All frames are **text** WebSocket frames whose body is a JSON object. Every
object has a `type` discriminator field.

```jsonc
// Generic envelope:
{
  "type": "<one of: handshake | event | storage | command>",
  ...                                 // type-specific fields
}
```

There is no version field. **(inferred — and a known weakness we'll likely
want to add in a future protocol revision, but not in this rewrite.)**

---

## 3. Direction: server → client

LoggerNext (the desktop app) is the **server**. It sends two message types.

### 3.1 `handshake`

Sent immediately after a client connection reaches `.ready`. Establishes a
session id.

```json
{
  "type": "handshake",
  "id": "550E8400-E29B-41D4-A716-446655440000"
}
```

- `id` — UUID generated by LoggerNext for this connection.

**(inferred — the SDK appears to accept this but does not echo it. Whether
the SDK uses the `id` for anything is unclear.)**

### 3.2 `command`

Sent in response to user action in the command bar or storages screen.

```json
{
  "type": "command",
  "command": "storage.list"
}
```

- `command` — opaque string interpreted by the SDK.

**Known commands the current desktop app sends:**

| Command         | Meaning                                                       | Sender                                |
|-----------------|---------------------------------------------------------------|---------------------------------------|
| `storage.list`  | Request a snapshot of session/local/keychain storage.         | Storages screen reload button         |
| `cmdlist`       | Request the list of commands the SDK currently supports.      | Command bar autocomplete refresh      |
| (anything else) | Free-form command text typed by the user in the command bar.  | Command bar submit                    |

**Open questions for the SDK team:**

1. **`storage.set` (or equivalent)** — for D6 (edit-and-push-back), we need
   a command to write a single key. Does it exist? If so:
   - Command name and full payload shape
   - Per-namespace support (can we write to keychain?)
   - Error response on failure
2. **Acknowledgements** — does the SDK confirm receipt of any command?
3. **Command discovery** — what does the SDK actually return in response
   to `cmdlist`? **(inferred to be an event-shaped message, but format
   undocumented.)**

## 4. Direction: client → server

The SDK sends two message types.

### 4.1 `event`

A single log event. The interesting wrinkle: the actual event payload is
**a JSON string nested inside the outer JSON envelope** (double-encoded).
This is preserved as-is for compatibility.

```json
{
  "type": "event",
  "id": "550E8400-E29B-41D4-A716-446655440000",
  "event": "{\"subsystem\":\"com.example.auth\",\"timestamp\":1715784000000,\"level\":\"info\",\"message\":\"login OK\",\"category\":\"network\",\"data\":{\"userId\":\"abc\"},\"context\":{\"build\":\"1.2.3\"}}"
}
```

- `id` — UUID; **(inferred — current desktop ignores it and generates its
  own UUID for the EventRecord. Worth keeping the SDK's id in v2.)**
- `event` — a JSON string. Decoded as follows:

#### 4.1.1 Inner event payload

```jsonc
{
  "subsystem":  "com.example.auth",      // required, string
  "timestamp":  1715784000000,           // required, milliseconds since epoch (UInt)
  "level":      "info",                  // required, string OR integer (see below)
  "message":    "login OK",              // required, string
  "category":   "network",               // optional, string (defaults to "")
  "data":       { ... },                 // optional, arbitrary JSON object
  "context":    { ... }                  // optional, arbitrary JSON object
}
```

**Level encoding (heterogeneous).** The SDK sends `level` as either a
string or an integer. The desktop app accepts both. Mapping:

| String      | Integer | Severity |
|-------------|---------|----------|
| `verbose`   | `0`     | lowest   |
| `debug`     | `1`     |          |
| `info`      | `2`     |          |
| `warning`   | `3`     |          |
| `error`     | `4`     | highest  |

LoggerNext normalizes both encodings to the string form on decode and
stores only the string in `event.level`.

**`data` and `context`.** Both are free-form JSON objects (any nesting,
any value types). LoggerNext stores them as opaque JSON blobs
(`event.data_json`, `event.context_json`) and renders them in the detail
pane via a recursive tree view.

**Validation rules LoggerNext applies on decode.**

1. Missing `subsystem`, `timestamp`, `level`, or `message` → reject the
   event. A synthetic event with subsystem `loggernext.protocol` and
   level `warning` is appended to the store instead, including the raw
   payload for debugging.
2. Unknown level value (string or integer outside the table above) →
   reject. Same handling as above.
3. Invalid JSON in the `event` string → reject. Same handling.

### 4.2 `storage`

A snapshot of the client's storage namespaces, sent in response to
`storage.list`.

```json
{
  "type": "storage",
  "id": "550E8400-E29B-41D4-A716-446655440000",
  "data": {
    "session": {
      "currentUserId": "abc",
      "featureFlags":  "{\"premium\":true}"
    },
    "local": {
      "lastLoginAt": "1715784000"
    },
    "secure": {
      "authToken": "...redacted..."
    }
  }
}
```

- `id` — UUID; **(inferred — purpose unclear, ignored by current
  desktop.)**
- `data` — object keyed by namespace. Known namespaces:
  - `session` — in-memory session storage
  - `local` — persistent local storage
  - `secure` — keychain (rendered in the UI as "keychain")

**Open questions for the SDK team:**

1. **Value type.** The Swift wire type declares
   `[String: [String: [String: String]]]` (three levels deep), but the
   consumer code treats values as opaque and walks them recursively
   regardless of depth. **What is the SDK's actual value shape?** A
   single-level `[namespace: [key: stringValue]]`? Or can values be
   objects/arrays/nested structures? This affects D6 (edit-and-push-back)
   significantly.
2. **Namespace stability.** Are there ever namespaces other than the
   three above? If the SDK supports custom namespaces, LoggerNext should
   render them generically.
3. **Snapshot vs delta.** Is this always a full snapshot, or might the
   SDK ever send only the diff? **(inferred to be always-full.)**

---

## 5. Encoding rules

- Charset: UTF-8.
- JSON keys: exact case as shown above. **(inferred — the current code
  uses exact-match `as? [String: Any]` lookups.)**
- Trailing commas: not allowed (standard JSON).
- Numbers: `timestamp` is a 64-bit unsigned integer (milliseconds).
  Decoded as `UInt64` to avoid overflow into the 2038-on-32-bit problem.

---

## 6. Connection lifecycle

1. Client opens WebSocket connection to `ws://<desktop-ip>:9080`.
2. LoggerNext (per D2: single-client) either:
   - Accepts: sends `handshake` immediately, opens a new session in the
     store, and starts forwarding inbound `event` / `storage` frames.
   - Rejects: closes with WebSocket close code **1008 (policy violation)**
     and reason text `"another client is already connected"`. The
     rejected client should not retry rapidly.
3. The connection remains open indefinitely. Either side may close at
   any time.
4. On unexpected close (network failure, client crash), LoggerNext marks
   the current session's `ended_at` and writes a synthetic event with
   subsystem `loggernext.transport`, level `info`, message
   `"connection closed (reason: …)"`.
5. New connections after a close start a new session.

## 7. Backwards / forwards compatibility

**Backwards.** LoggerNext must decode every event the current Logger
desktop can decode. Test fixtures captured from real client sessions are
checked into `LoggerNextTests/Fixtures/`.

**Forwards.** Unknown top-level `type` values are rejected with a logged
synthetic event but do not break the connection. Unknown fields inside
known payloads (e.g., a new optional field on the inner event) are
ignored silently.

This gives the SDK headroom to add new message types and fields without
breaking LoggerNext, at the cost of LoggerNext not surfacing them until
explicitly updated.

---

## 8. What LoggerNext does *not* do

- **Does not authenticate.** Anyone on the local network can connect.
  Same as today. (If this becomes a concern, an opt-in shared-secret
  header is the cheapest addition; out of scope for v1.)
- **Does not throttle.** No flow control. The SDK can send as fast as
  it likes; the LogStore append batch absorbs bursts.
- **Does not request retransmission.** If a frame is malformed, the
  data is lost (apart from the synthetic error event recording that it
  happened).

## 9. Test fixtures to capture

To validate the decoder against reality, capture and check in:

- One real `event` frame at each log level (5 fixtures).
- One real `event` frame with non-trivial `data` and `context` payloads.
- One real `event` frame using **integer** level encoding (if the SDK
  ever sends this).
- One real `storage` snapshot from each platform target (iOS, tvOS).
- One example of every command the SDK accepts (for round-trip testing
  of the outbound path).

These should live as raw JSON files under
`LoggerNextTests/Fixtures/wire/` and be replayed by parameterized
`ProtocolDecoderTests`.
